# Windows 开始菜单顶层快捷方式发现设计

## 状态

- 日期：2026-07-22
- 状态：设计已批准，等待实施计划
- 实现基线：本地 `main@16dd26ea9346809c1aad8462aa811df37036b68b`
- 影响范围：Rust Start Menu Known Folder 查询、快捷方式候选收集、应用身份与发现测试

## 背景与目标

当前应用发现只递归扫描 `FOLDERID_Programs` 与 `FOLDERID_CommonPrograms`。部分经典 Win32 安装器把主入口直接写在 `FOLDERID_StartMenu` 或 `FOLDERID_CommonStartMenu` 根目录，而不是对应的 `Programs` 子目录。例如企业微信主入口位于公共 Start Menu 根目录，现有扫描不可达；AppsFolder 中同名项又使用 `Microsoft.AutoGenerated.{...}` 标识，不能通过 packaged AUMID 校验，因此主应用不会进入快照。

本设计补充以下两个来源：

```text
user-top-level   = FOLDERID_StartMenu 根目录直属普通 .lnk
common-top-level = FOLDERID_CommonStartMenu 根目录直属普通 .lnk
```

现有两个来源保持不变：

```text
user   = FOLDERID_Programs，递归扫描
common = FOLDERID_CommonPrograms，递归扫描
```

目标是让企业微信这类顶层主入口沿用现有桌面快捷方式链进入搜索、图标和启动流程。前端、协议、排序状态机、缓存结构和 AppsFolder packaged 规则均不改变。

## 与既有规格的关系

本规格只修订 `2026-07-18-start-menu-application-discovery-design.md` 中“生产扫描器只能查询两个 Known Folder”及对应两 GUID 测试条款，将生产来源扩展为本文固定的四个 Known Folder。旧规格关于 `FOLDERID_Programs`、`FOLDERID_CommonPrograms` 递归扫描、`user/common` 身份、shortcut 解析、安全、诊断和 DTO 的其余合同继续有效。

`2026-07-21-windows-appsfolder-discovery-design.md` 的 AppsFolder 枚举、packaged AUMID 验证和启动合同不变；执行顺序调整为四个 Start Menu 根完整成功后再枚举 AppsFolder。`2026-07-21-launcher-application-icons-design.md` 的后台图标提取、资源所有权和前端安全投影合同也不变。

## 方案选择

### 采用：额外查询两个 Start Menu Known Folder，只扫描根目录直属 `.lnk`

生产扫描依次查询 `FOLDERID_Programs`、`FOLDERID_CommonPrograms`、`FOLDERID_StartMenu`、`FOLDERID_CommonStartMenu`。前两个根继续递归，后两个根只检查 `read_dir(root)` 返回的直属条目。顶层根中的任何目录，包括 `Programs`，都不进入递归。

这条路径复用现有 `.lnk` 信任边界、启动动作和图标提取，代码增量最小；同时不会把 `Programs` 子树扫描第二次。

### 拒绝：从 Start Menu 根递归扫描并在运行时去重

`FOLDERID_StartMenu` 与 `FOLDERID_CommonStartMenu` 包含各自的 `Programs` 子目录。递归扫描会重复现有来源，再依赖路径或名称去重，既改变身份边界又扩大错误和性能面。

### 拒绝：接受 AutoGenerated AppsFolder 标识或改扫 Desktop/注册表

`Microsoft.AutoGenerated.{...}` 不是 `ParseApplicationUserModelId` 可验证的 packaged AUMID，不能进入现有 `IApplicationActivationManager` 启动目标。Desktop、卸载注册表、Windows Search index、网页和文件系统任意位置也不属于本设计；它们会引入新的信任、身份和启动合同。

## Known Folder 与扫描边界

四个 Known Folder 查询全部通过现有 `SHGetKnownFolderPath` 路径执行，固定使用：

```text
flags  = KF_FLAG_DONT_VERIFY
hToken = NULL
```

不读取 `%APPDATA%`、`%ProgramData%` 或其他环境变量作为 fallback，不接受前端、配置、命令行或注册表提供根路径。每次调用返回的非空 `PWSTR` 无论后续成功或失败都由现有所有权守卫调用 `CoTaskMemFree`；空指针同样走安全的统一析构路径。

查询顺序固定为：

1. `FOLDERID_Programs`
2. `FOLDERID_CommonPrograms`
3. `FOLDERID_StartMenu`
4. `FOLDERID_CommonStartMenu`

任一 Known Folder 查询失败都返回现有 `KnownFolderQuery`，整次发现失败。查询成功但根不存在时，该根贡献零项并增加现有 `missing_roots`；根存在但不是目录、根不可读取或根自身为 reparse point 时，分别沿用 `RootNotDirectory`、`RootUnavailable`、`RootReparsePoint` 硬错误，整次发现失败。

### Programs 根

`FOLDERID_Programs` 和 `FOLDERID_CommonPrograms` 继续使用现有递归 `read_dir` 行为。现有相对路径规范化、reparse 拒绝、普通 `.lnk` 过滤、诊断计数和错误语义不得改变。

### 顶层根

`FOLDERID_StartMenu` 和 `FOLDERID_CommonStartMenu` 只处理根目录一次 `read_dir` 返回的直属条目：

- 先用 `symlink_metadata` 检查每个直属条目。
- 无法读取元数据时跳过该项并增加 `inaccessible_entries`。
- 直属条目带 `FILE_ATTRIBUTE_REPARSE_POINT` 时跳过并增加 `reparse_entries`；不跟随、不解析目标。
- 直属目录无论名称为何都直接忽略，不递归，也不增加诊断；这明确包含 `Programs` 目录。
- 只接受 `metadata.is_file()` 且扩展名以 ASCII 大小写不敏感方式等于 `.lnk` 的普通文件。
- 文件名和相对组件必须无损转换为 Unicode；失败时跳过并增加 `non_unicode_entries`，禁止 lossy 文本。
- 显示名称仍是 `.lnk` 文件名去掉扩展名，不读取 Shell 描述或目标名称。
- 根目录 `read_dir` 本身失败是 `RootUnavailable` 硬错误，不降级为部分成功。

顶层根不会扫描其任何子目录，因此不会触达 `Programs`、OEM 文件夹、用户自建文件夹或它们内部的 shortcut。

## 身份与兼容性

现有 Programs 身份预映像必须逐字保持不变：

```text
"start-menu-v1\0user\0" + normalizedRelativeShortcutPath
"start-menu-v1\0common\0" + normalizedRelativeShortcutPath
```

新顶层 scope 使用同一 CNG SHA-256 和 `app-` 加 64 位小写十六进制输出，但固定使用两个新身份字节：

```text
"start-menu-v1\0user-top-level\0" + normalizedFileName
"start-menu-v1\0common-top-level\0" + normalizedFileName
```

`normalizedFileName` 是直属 `.lnk` 的单个普通相对组件，保留 `.lnk` 扩展名后执行现有 Unicode `to_lowercase`。不修剪、不 canonicalize、不展开环境变量。

该方案可以在现有最小 `RootKind` 表示中保留 `User`、`Common` 及其 identity 不变，只为顶层增加两个明确 scope；不引入通用 source framework、配置化扫描器或动态身份字符串。

身份预映像不得包含绝对根路径、Known Folder GUID、显示名称、快捷方式目标、参数、图标、文件元数据或枚举顺序。由此保证：

- 所有既有 Programs `appId` 固定向量完全不变。
- 用户顶层与公共顶层的同名入口身份不同。
- 顶层与 Programs 的同名入口身份不同。
- 同一顶层 scope 内仅大小写变化保持身份；其他重命名或跨 scope 移动产生新身份。
- 不按名称、目标或显示文本去重；真正重复 `appId` 仍使整次发现失败。

## 快捷方式、图标与启动复用

顶层候选进入现有 `load_shortcut`：

- `IPersistFile::Load` 失败只跳过该快捷方式并增加 `invalid_shortcuts`。
- COM 不可用仍是整次发现的 `ComUnavailable` 硬错误。
- `SLGP_RAWPATH`、`executable` 词法验证和“不检查目标存在性”合同不变。
- 无可用 executable 映射的有效快捷方式仍保留，并增加 `unmapped_executables`。
- 图标继续在后台扫描线程从可信 `.lnk` Shell item 尽力提取；失败只得到 `icon = None`，不删除应用。

顶层入口仍生成 `ApplicationLaunchTarget::Shortcut`，通过现有 `registry_entry` 得到“应用程序”副标题，并由现有 `ResultAction::LaunchApplication` 私有保存 `.lnk` 与 executable。前端仍只收到标题、副标题、可选安全图标和不透明结果 ID；路径与目标不序列化。

启动继续使用可信 `.lnk`：先按现有 executable 尽力激活窗口，否则 `ShellExecuteW` 启动 shortcut。无需新增动作变体、命令或 fallback。

## 合并、缓存与错误事务

四个 Start Menu 根在同一次后台 `discover()` 中生成一个完整桌面 shortcut 快照，再与有效 packaged AppsFolder 快照合并。只有所有 Known Folder 查询、所有根扫描、COM 初始化、AppsFolder 绑定与枚举、身份计算及冲突检查都成功，`AppCache::refresh_with` 才一次性替换旧 `Vec<Application>`。

任一新增顶层根的查询或扫描硬失败，必须与现有根失败一样拒绝整次 refresh 并保留上次完整快照；不得发布仅含 Programs 或仅含一个顶层根的部分快照。逐项软错误只更新现有定长聚合计数，不增加路径、名称、HRESULT 或新诊断字段。

初始 refresh 与手动 rescan 继续在现有后台线程/`spawn_blocking` 路径运行。新增工作最多是两个非递归 `read_dir` 和直属有效 `.lnk` 的现有解析/图标提取；不增加线程、队列、锁或缓存。托盘显示、WebView 和搜索快照读取不等待扫描完成。

## 排序与企业微信行为

不改变 `rank`。匹配等级仍按 exact、prefix、contains、subsequence 排序，并先于 alias、`use_count`、名称、入口类型和 `appId`。

因此查询“企业微信”时：

- 顶层主入口“企业微信”是 exact match。
- Programs 内“卸载企业微信”是 contains match。
- 主入口自然排在卸载入口之前，即使卸载入口有更高 `use_count`。

发现层不按产品名特判，不屏蔽“卸载”文本，不按名称合并两个入口，也不增加企业微信专属别名或名单。

AppsFolder 继续只接受 `ParseApplicationUserModelId` 完整成功的 packaged AUMID。`Microsoft.AutoGenerated.{...}` 仍按无效 AUMID 跳过并增加 `invalid_packaged_aumids`，不得因同名顶层 shortcut 已可发现而放宽 validator。

## 隐私与安全

- 新来源仍由 Windows Known Folder API 决定，不允许任意路径输入。
- 顶层扫描不递归目录，将读取面限制为两个根的直属普通 `.lnk`。
- reparse 根是硬错误，reparse 子项一律跳过，不访问目标。
- 不 canonicalize shortcut 或 executable，不跟随目录链接，不读取目标文件内容。
- shortcut 路径、目标路径、参数、绝对根和原生错误仍只存在于 Rust 可信边界。
- 日志、错误、诊断与前端 DTO 不得包含应用名称、路径、AUMID、GUID 或 HRESULT 文本。
- 不从 Desktop、卸载注册表、Windows Search index、AppX manifest、网页或远程资源补入口。

## 测试合同

实施必须按 TDD 建立以下最小门禁：

1. 生产 Known Folder provider 以固定顺序调用四个准确 GUID；每次均使用 `KF_FLAG_DONT_VERIFY` 与 null token。成功、失败、空指针和无效 Unicode 路径均证明返回内存按现有所有权规则释放。
2. 在注入的用户/公共顶层临时根直属位置创建普通 `.lnk`，先取得现有实现无法发现的 RED，再实现后断言两项进入快照且目标变体为 `Shortcut`。
3. 在顶层临时根创建 `Programs\Nested.lnk` 和其他子目录 shortcut，断言均不进入候选；同一 fixture 的直属 `.lnk` 仍进入。
4. 现有 `FOLDERID_Programs` 与 `FOLDERID_CommonPrograms` 注入根继续递归发现嵌套 `.lnk`，现有排序、损坏项、missing root 与 Start Menu boundary 测试保持通过。
5. 为新 scope 固定 app ID 已知向量；断言相同文件名在 user top-level、common top-level、user Programs、common Programs 四个 scope 的 ID 全部不同，并重跑旧 `User/Common` 固定向量保证零变化。
6. 顶层根不存在增加 `missing_roots`；根不可读、非目录、根 reparse 分别返回既有硬错误；直属 reparse 文件/目录跳过并增加 `reparse_entries`；直属元数据读取失败增加 `inaccessible_entries`；无损 Unicode 失败增加 `non_unicode_entries`。
7. 顶层有效 `.lnk` 的 invalid shortcut、无 executable 映射、图标成功/失败和 DTO 隐私继续由现有 loader/icon/registry 测试覆盖，不复制平台实现测试。
8. 用“企业微信”顶层主入口和“卸载企业微信”Programs 入口组成同一应用切片，断言查询“企业微信”时主入口先于卸载入口；不添加产品名分支。
9. 新增任一 Known Folder 查询失败或顶层根扫描硬失败时，`AppCache` 保留上次完整快照；AppsFolder 失败事务测试继续通过。
10. characterization 测试继续证明 `Microsoft.AutoGenerated.{...}` 无法通过 packaged AUMID validator，计算器/设置等有效 AUMID 行为不变。
11. `cargo fmt --all -- --check`、Rust library tests、`cargo check --lib`、两组 warnings-as-errors Clippy、all-features、前端 tests/build、安全配置与 Start Menu boundary 脚本全部通过。

Windows 人工验收：

1. 确认系统存在一个用户或公共 Start Menu 根目录直属主应用 `.lnk`，且 `Programs` 中可同时存在同名或卸载入口。
2. 启动 UiPilot 并等待后台扫描完成，输入主应用精确名称；主入口必须出现并排在包含该名称的卸载入口前。
3. 启动主入口，确认执行主应用 `.lnk`，不是卸载入口。
4. 搜索一个既有嵌套 Programs 应用以及计算器/设置，确认递归 shortcut、packaged AppsFolder、图标和启动均无回归。
5. 立即通过托盘打开主界面，确认新增扫描不改变现有非阻塞展示行为。

## 风险与约束

- 四个 Known Folder 形成同一严格事务，因此新增根损坏会使 refresh 保留旧快照；这是与现有根一致的 fail-closed 选择，不提供部分结果降级。
- 顶层 scope 必须拥有独立身份字节；复用旧 `user/common` 会让同名顶层与 Programs 入口产生相同 ID，属于不可接受的兼容性错误。
- Start Menu 根含 `Programs` 子目录；任何递归实现都会重复扫描和扩大安全面，因此 direct-only 必须由测试锁定。
- 新增两次非递归目录读取的成本有明确上界；Shell 图标提取仍可能慢，但只发生在现有后台扫描线程，不允许移动到 UI 路径。

## 非目标

- 不扫描 Desktop、公共 Desktop、卸载注册表、Windows Search index、任意磁盘目录、AppX manifest、网页或文件内容。
- 不接受或启动 `Microsoft.AutoGenerated.{...}` AppsFolder 项，不增加 Shell item 启动目标。
- 不递归 `FOLDERID_StartMenu` 或 `FOLDERID_CommonStartMenu` 的任何子目录。
- 不改变既有 Programs app ID、packaged app ID、名称去重、排序、别名、`use_count`、结果上限或启动动作。
- 不屏蔽卸载项，不维护产品名单，不为企业微信添加特判。
- 不增加前端字段、命令、组件、设置、配置项、依赖、线程、缓存或通用发现框架。
- 不在本规格阶段编写实施计划、生产代码或测试。

## 完成标准

- 四个 Known Folder 查询与 direct/recursive 边界无歧义，旧 Programs 行为和身份完全兼容。
- 顶层两个 scope 使用固定、独立且可审计的预映像，绝对路径与展示数据不进入 ID。
- 企业微信类顶层主入口可通过既有 shortcut、图标、结果注册、排序和启动链自然工作。
- AutoGenerated AppsFolder、Desktop、注册表、Windows Search 与网页明确保持范围外。
- 错误、诊断、隐私、缓存事务、后台线程和人工验收均有可执行测试合同。
