# 主窗口拖动与位置持久化设计

## 状态

- 日期：2026-07-22
- 状态：设计已确认，等待规格复核
- 实现基线：本地 `main@d912f0bad8f2bf9ff38613ae0b59b482b4fd1d17`

## 背景

主窗口当前无系统标题栏且不可调整大小，用户无法移动窗口。每次显示窗口时，生命周期逻辑都会调用居中，因此即使窗口位置发生变化也不会保留。

本次增加最小的窗口拖动和位置持久化能力：用户按住非交互空白区域即可拖动窗口；窗口隐藏后再次显示或程序重启后，窗口恢复到上次消失时的位置。

## 已确认决策

| 项目 | 决策 |
| --- | --- |
| 可拖动区域 | 非交互空白区域 |
| 不可拖动区域 | 输入框、按钮、结果项、滚动区域及其他交互控件 |
| 保存时机 | 窗口消失时保存一次，不在拖动过程中持续写盘 |
| 恢复时机 | 每次显示主窗口前 |
| 无保存位置 | 主屏幕居中 |
| 保存位置失效 | 主屏幕居中 |
| 实现约束 | 复用现有 `SettingsStore` 和 Tauri 原生能力，不新增依赖 |

## 目标

1. Launcher 与设置界面的非交互空白区域支持按住并拖动主窗口。
2. 所有现有隐藏入口都在窗口消失时记录最后位置。
3. 所有现有显示入口都优先恢复已保存位置，不再无条件居中。
4. 应用重启后仍能恢复位置。
5. 显示器拔插、缩放或分辨率变化导致位置不可用时，窗口安全回退到主屏幕居中。

## 非目标

- 不保存窗口大小、最大化、最小化或置顶状态。
- 不按显示器分别保存多个位置。
- 不增加窗口位置设置项或“重置位置”按钮。
- 不监听每一个移动事件，不引入防抖任务或后台位置服务。
- 不改变窗口固定尺寸、无边框和不可调整大小的现状。

## 交互设计

前端在实际空白表面标记 Tauri 原生拖动区域。标记只覆盖空白容器本身，不覆盖输入框、按钮、结果行和滚动内容，因此现有点击、选择、键盘输入与滚动行为保持不变。

“长按移动”采用桌面窗口的标准按住拖动语义：按下鼠标并移动即开始拖动，不额外增加长按计时器、拖动手柄或视觉提示。

Launcher、设置页以及状态区域共享同一主窗口。切换视图不改变已保存位置，也不触发位置写盘；只有窗口实际隐藏时才保存。

## 持久化模型

在现有设置模型中增加一个可选的内部窗口位置：

```text
windowPosition?: {
  x: integer,
  y: integer
}
```

坐标使用 Tauri 返回的物理屏幕坐标，与窗口和显示器位置 API 保持同一坐标系。字段使用 Serde 默认值，因此旧 `settings.json` 缺少该字段时正常加载，并按“无保存位置”处理。

窗口位置不是设置页可编辑字段，不加入前端设置加载或保存协议。`SettingsStore` 提供窄范围更新，只替换窗口位置并复用现有原子写盘流程，避免设置页保存覆盖该内部字段。

## 隐藏流程

现有隐藏入口继续汇聚到共享的清理与隐藏路径。该路径按以下顺序执行：

1. 在窗口仍可查询时读取当前外部位置。
2. 执行现有结果清理与窗口隐藏。
3. 若位置读取成功，则通过 `SettingsStore` 原子持久化该位置。

位置读取或持久化失败不得阻止窗口隐藏，也不得改变现有结果清理语义。失败只意味着下次显示可能使用上一次成功保存的位置或回退居中。

该流程覆盖失焦隐藏、执行结果后隐藏、关闭请求转隐藏及其他现有共享隐藏调用方。实现前必须核对所有直接 `window.hide()` 调用，确保不存在绕过共享路径的主窗口隐藏入口。

## 显示流程

所有快捷键、托盘和单实例唤起继续汇聚到现有主窗口显示路径。显示前执行：

1. 从 `SettingsStore` 读取可选位置。
2. 获取当前可用显示器工作区域、主显示器及主窗口固定尺寸。
3. 若保存位置能让完整窗口落在某个可用显示器工作区域内，则设置该位置。
4. 若无保存位置、位置不再有效或位置设置失败，则在主显示器工作区域内居中；主显示器信息也不可用时才调用现有居中逻辑。
5. 按现有顺序继续设置置顶、显示、聚焦和发送 shown 事件。

完整窗口必须位于某个可用显示器工作区域内才视为有效。这一保守规则避免无标题栏窗口恢复到无法操作的位置；跨显示器摆放不作为 MVP 支持场景。

## 错误处理

- 读取当前位置失败：继续隐藏，不更新已保存位置。
- 写入设置失败：继续隐藏，保留内存或磁盘中最后一次成功保存的位置。
- 枚举显示器失败：本次显示优先在主显示器居中，主显示器查询也失败时回退到现有居中逻辑。
- 保存位置无效：不修改持久化值，本次显示在主显示器居中；用户再次移动并隐藏后自然覆盖旧值。
- 设置位置失败：本次显示在主显示器居中，显示与聚焦流程继续执行。

窗口位置是便利性状态，不得让显示、隐藏、执行或退出等主要流程失败。

## 测试

自动化测试保持聚焦：

- 设置模型兼容缺少 `windowPosition` 的旧 JSON，并能原子更新位置且保留其他设置。
- 共享隐藏逻辑在读取位置成功时保存一次；读取或保存失败仍执行隐藏。
- 显示逻辑在位置有效时恢复，在无位置、位置失效、显示器查询失败或设置位置失败时居中。
- 现有显示顺序、结果清理和隐藏失败语义保持不变。
- 前端测试确认交互控件未被标记为拖动区域。

人工验收：

1. 从 Launcher 空白区域按住拖动，窗口跟随移动；输入框、结果项和按钮仍可正常操作。
2. 将窗口移动到屏幕内任意位置，触发失焦隐藏，再通过快捷键显示，位置保持。
3. 移动窗口后退出并重启程序，再次唤起，位置保持。
4. 在设置页空白区域拖动，设置控件仍可编辑和点击。
5. 保存位置后拔掉对应显示器或降低分辨率，再次显示时窗口出现在主屏幕中央。

## 拒绝方案

- **持续监听窗口移动并写盘：拒绝。** 会产生频繁磁盘写入，并需要防抖和退出竞态处理；窗口消失时保存一次已经满足需求。
- **引入窗口状态插件：拒绝。** 当前只需要两个坐标，现有设置存储和 Tauri API 足够；插件还会引入尺寸、状态等未要求语义。
- **自定义鼠标长按计时和拖动算法：拒绝。** Tauri 原生拖动区域已覆盖标准桌面交互，自定义实现只会增加前端状态和边界问题。
- **无条件恢复保存坐标：拒绝。** 显示器布局变化后可能让无边框窗口落到屏幕外且无法找回。
