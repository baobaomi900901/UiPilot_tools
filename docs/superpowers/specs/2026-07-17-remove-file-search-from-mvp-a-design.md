# 文件搜索移出 MVP-A 决策设计

## 状态

- 日期：2026-07-17
- 状态：已批准
- 影响范围：MVP-A 规格与 Windows Launcher Foundation 计划

## 背景

SystemIndex Spike 的功能查询、索引范围和服务关闭行为符合预期，但 ProcMon 将宿主发起的目录查询和非允许文件读取归因到 Spike 进程，违反“宿主不触发文件系统遍历或文件内容读取”的硬门槛。完整证据见 [SystemIndex Spike 结果](../../spikes/2026-07-17-systemindex-results.md)。

MVP-A 的首要目标是验证键盘启动器能否形成持续的应用启动和激活使用。文件搜索不是验证该假设的必要条件。

## 决策

MVP-A 删除所有文件搜索能力，不委托 Windows Explorer 搜索，也不保留禁用路由、功能开关、占位界面、计数或测试。主输入框对所有非空输入执行应用搜索；MVP-A 不定义斜杠命令系统。

MVP-A 保留以下产品范围：

1. 全局快捷键唤起启动器。
2. 应用发现、搜索和启动。
3. 对已运行应用执行尽力激活，并在系统拒绝或映射不唯一时回退到启动。
4. 基础设置、开机启动、签名安装包和发布验证。

每日 5 次成功动作门槛保持不变。成功动作只由应用启动请求和激活成功组成，因此该门槛明确验证纯应用启动/激活使用频率。

## 被拒绝的方案

- 继续 SystemIndex 生产路线：实证已违反 I/O 硬门槛，不能通过重复采集解除阻塞。
- 委托 Windows Explorer 搜索：宿主无法稳定控制结果语义、完成状态和任务成功率，降低 MVP-A 的可验证性。

## 文档合同

- 产品规格只保留一条带报告链接的历史 No-Go 记录；生产需求、路由、计数、性能指标、错误处理和验收测试不保留文件搜索合同。
- Foundation 计划不保留文件字段、禁用提示或斜杠命令分支。
- Foundation 完成不等于 MVP-A；下一必需交付物是签名安装包与发布验证计划。
- 文件搜索降为未承诺的后续候选。重新进入范围前必须另写 PRD 和技术评审，不得复用失败路线作为默认方案。

## 非目标

本决策不选择新的文件搜索架构，不编写文件搜索代码或实施计划，也不改变翻译、macOS 和第三方插件仍在 MVP-A 之外的结论。
